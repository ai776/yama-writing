# book-feedback-ingest — ⑩ フィードバック取込

編集者・校閲者のフィードバックを構造化し、book-rewrite が効率的にリライトできる修正指示書に変換する。

## なぜ必要か

フィードバックが生データのままだと:
- 修正ポイントが散在して優先順位が不明
- 同じ指摘が複数箇所にあっても統合されない
- book-rewrite がフィードバックの意図を誤解するリスク
- 修正の粒度がバラバラ（全体的な方針指摘 vs 一字修正が混在）

構造化された修正指示書に変換することで:
- 修正の優先順位が明確になる
- 章ごと・カテゴリごとに整理される
- book-rewrite が迷わず修正に着手できる

## Inputs / Outputs

- **In:** フィードバック原文（修正ポイント原稿、レビューコメント、メール、音声メモ文字起こし等）、`BookSpec.json`、原稿
- **Out:** `feedback_spec.md`（構造化された修正指示書）

## ワークフロー

1. **フィードバック原文の読み込み** — 全てのフィードバック素材を読み込む
2. **フィードバックの分類:**
   - **全体方針** — トーン変更、読者層の見直し等、書籍全体に関わる指摘
   - **章単位** — 特定の章に対する指摘（論理、内容、構成）
   - **箇所単位** — 特定の文・段落に対する指摘（表現、事実、数値）
   - **形式** — 誤字脱字、表記統一、フォーマット
3. **優先度の判定:**
   - MUST — 必ず修正（事実誤認、論理破綻、著者の明確な修正指示）
   - SHOULD — 強く推奨（読みやすさ改善、論調調整）
   - COULD — あれば良い（好みの範囲、微修正）
4. **修正意図の明確化** — 「何を」「なぜ」「どう直すか」を三点セットで記述
5. **原稿との紐付け** — 各修正指示を原稿の章・節・行番号と紐付ける
6. **重複・矛盾の統合** — 同一箇所への複数指摘を統合、矛盾する指摘はフラグ
7. **レビューチェーン:**
   - book-qa に提出（G10-Q: 分類の正確性、網羅性）
   - book-producer に提出（G10-P: 修正方針の妥当性）
   - 著者に提出（G10-H: 修正方針の承認）← 特に全体方針の変更時は必須
