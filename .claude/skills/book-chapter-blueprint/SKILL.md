# book-chapter-blueprint — ⑦ 章設計図

各章の「設計図」を作成する。見出し（book-heading）の前工程であり、
**見出しは設計図の結果であって、出発点ではない**。

## なぜ Blueprint が必要か

見出しから直接本文を生成すると:
- 「株・NISAではダメな理由」→ つい批判的な論調に引きずられる
- 「出口戦略が最重要」→ 他の要素が軽視される
- 章の核心メッセージが曖昧なまま執筆が始まる

Blueprint で設計を固めてから見出しを作ることで:
- 章の意図が明確になり、見出しも本文も一貫する
- 禁止事項を先に定義するので論調のブレを防止できる
- **人間チェックを挟むタイミングが明確になる**

## Inputs / Outputs

- **In:** `BookSpec.json`, `toc.md`, `chapter_budget.md`, `knowledge_base.md`（あれば）
- **Out:** `chapter_blueprints.md`（全章のBlueprint集）

## ワークフロー

1. **BookSpec.json 確認** — 核心メッセージ・ペルソナ・USPを把握
2. **toc.md 分析** — 承認済み目次の章構成・各章のテーマを確認
3. **knowledge_base.md 参照**（あれば）— 各章に紐付くナレッジ項目を確認
4. **各章の Blueprint 作成:**
   - 章の核心（1文）
   - 読者の現状認識
   - 本章で壊す誤解
   - 本章のUSP（独自の価値提案）
   - 優先順位ルール（1〜3位）
   - 禁止事項
   - 必要要素（具体例・図表・数値・ナレッジID等）
   - 章のストーリー構造（①〜⑥のステップ）
5. **章間の整合性チェック** — USPの重複なし、禁止事項の矛盾なし
6. **3段階レビューチェーン:**
   - book-qa に提出（G7-Q: Blueprint間の整合性、BookSpec整合性）
   - book-producer に提出（G7-P: 戦略整合性、読者体験の設計）
   - **著者に提出（G7-H: Blueprint の最終承認）** ← 人間チェック必須

**重要:** Blueprint 承認後に book-heading を実行すること。逆順は禁止。

## Blueprint 各フィールドのルール

### 章の核心（1文）
- この1文だけ読めば章の存在意義がわかること
- NG: 「〜について説明する」（説明行為の記述は核心ではない）
- OK: 「不動産だけが銀行レバレッジを使える投資である」

### 読者の現状認識
- この章を読む前の読者が「今こう思っている」状態
- ペルソナの知識・先入観・不安を具体的に書く

### 本章で壊す誤解
- 読者の現状認識のうち、この章で覆す部分
- 「〜という思い込み」の形式で書く

### 本章のUSP
- この章にしかない独自の切り口・価値
- 他の章や他の書籍では語られていない角度

### 優先順位ルール
- 本文で強調すべき要素の優先順序（1〜3位）
- book-draft がどこに紙幅を割くべきかを制御する

### 禁止事項
- この章で絶対にやってはいけないこと
- 見出しの文言に引きずられやすい方向を明示的に禁止

### 必要要素
- この章に必ず含めるべき具体的な要素
- knowledge_base.md の該当ナレッジID（あれば）

### 章のストーリー構造
- 章内の情報提示順序を①〜⑥のステップで記述
- 基本パターン: ①誤解解除 → ②核心提示 → ③理由 → ④具体例 → ⑤行動含意 → ⑥次章への橋渡し
