# book-knowledge — ④ ナレッジ生成

cleaned_transcript.md（book-ingest 出力）から、AIが参照しやすい構造化ナレッジを生成する。

## なぜ必要か

生データ（文字起こし等）をそのまま参照すると:
- 長文すぎてAIの参照精度が落ちる
- 本質的な主張と雑談が混在する
- 後工程（book-draft）で情報の取捨選択に時間がかかる

構造化ナレッジに変換することで:
- AIが「主張」「根拠」「具体例」を正確に参照できる
- book-draft が必要な情報を即座に引き出せる
- 章ごとの情報マッピングが容易になる

## Inputs / Outputs

- **In:** `BookSpec.json`, `cleaned_transcript.md`（book-ingest 出力）
- **Out:** `knowledge_base.md`（構造化ナレッジ）

## ワークフロー

1. **BookSpec.json 確認** — 核心メッセージ・ペルソナ・情報源タイプを把握
2. **cleaned_transcript.md 読み込み** — 整形済みテキストを全文読み込み
3. **セグメント分割** — テーマ・話題の切り替わりを検出し、意味のまとまりに分割
4. **タグ付き抽出** — 各セグメントから以下のタグで本質を抽出:
   - `[主張]` — 著者の結論・意見・立場
   - `[理由]` — 主張の根拠・論理
   - `[具体例]` — 実例・ケーススタディ
   - `[数値]` — 数字・前提条件・計算根拠
   - `[反論]` — 想定される反論・その反駁
   - `[比喩]` — 著者独自の比喩・たとえ話
   - `[体験談]` — 著者自身の経験・エピソード
   - `[感情トーン]` — その箇所の感情的温度（熱い主張/冷静な分析/共感的 等）
   - `[思想軸]` — 著者の価値観・哲学が表出している箇所
5. **重複統合** — 同一テーマの複数箇所を統合（出現箇所は全て記録）
6. **章マッピング** — toc.md が存在する場合、各ナレッジ項目を該当章に紐付け
7. **レビューチェーン:**
   - book-qa に提出（G4-Q: 情報抽出の正確性・網羅性チェック）
   - book-producer に提出（G4-P: 戦略整合性・核心メッセージとの対応確認）
   - 著者に提出（G4-H: ナレッジ内容の承認）

## 抽出ルール

- **情報源に忠実:** 著者の発言・記述にない解釈を加えない
- **要約ではなく抽出:** 原文のニュアンスを保ったまま構造化する
- **出典を明記:** 各タグ付き項目に原文の該当箇所（行番号 or タイムスタンプ）を記録
- **著者の特徴的表現を保存:** 比喩・口癖・独自の言い回しは原文のまま記録
- **感情トーンの記録:** 著者が特に力を込めている箇所を検出し記録
